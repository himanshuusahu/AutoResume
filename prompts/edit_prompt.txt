## Experience bullet writing (quality rules)
When rephrasing each role’s bullets:
- Prefer **impact-first** ordering: start with the action (verb), then scope, then result.
- Add **quantified outcomes** when they are implied by the existing bullet or can be reasonably inferred from it (e.g., “reduced X by Y”, “improved coverage to Z%”, “cut cycle time”, “increased adoption”, “achieved N+”). If no metric exists in the current bullet, do not fabricate numbers—keep it outcome-focused instead.
- Make bullets **specific** (what you did, what tech/process you used, and what changed).
- Include **job-relevant keywords** inside the sentence (e.g., “CI/CD”, “Jenkins”, “Docker”, “REST APIs”, “SQL”, “Selenium/Playwright”, “testing”, “automation”, “monitoring”) only when they align with the current content.
- Keep each bullet **concise** (one strong idea). Avoid long sentences.
- Preserve the **meaning** of the original bullet; you may enhance clarity and impact but must not introduce unsupported claims.

## Technical Skills tailoring rules
Inside the existing skills itemize block:
- Use only the provided category labels (e.g., Languages/Frameworks/Developer Tools/etc.) and keep the same general block structure.
- Rebuild the skill lists so they are **most relevant to the job description**, prioritizing the extracted keywords.
- Maintain authenticity: do not introduce skills that are not present in the current skills section or clearly supported by the Experience section.
- Prefer consistent, ATS-friendly names (e.g., “CI/CD”, “PostgreSQL”, “Docker”, “Jenkins”, “Kubernetes”, “REST APIs”, “SQL”, “TypeScript”, “GitHub Actions” if present).
- Remove weaker/less relevant skills to make room for stronger job-matched ones.
